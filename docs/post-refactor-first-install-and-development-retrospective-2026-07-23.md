## 一、最终结论

本轮系统性重构可以收尾。

- WordPress-first 的 P0–P5 工程重构已经完成，不再把“继续清理架构”作为
  下一阶段主线。
- Cloud 与 WordPress 的责任边界已经稳定：Cloud 提供托管运行时，
  WordPress 保持权限、审阅、应用、最终写入和本地审计真相。
- 文本建议和图片往返均已形成可验证闭环；媒体 ACK 只证明传输完成，
  不代表 Cloud 获得 CMS 写入权。
- Cloud 已建立一次性首次安装、外部 PostgreSQL 18、RDS CA、
  管理员密钥和密钥轮换基础。
- 当前仍没有真实用户，已有结果证明工程路径可运行，但不等于真实用户价值、
  GA 容量或长期稳定性已经得到证明。
- Typecho、Z-BlogPHP、Ghost 和更多媒体类型只保留干净接缝，不在当前阶段
  同时实现。
- 下一阶段应转向 CVE 发布门槛、一次有边界的 WordPress 体验验证和
  GA 决策，不再发起新的结构性重构。

截至本文记录时，本分支还包含一组尚待后续发布的管理员体验修正：
Provider 连接标识自动生成、模型目录证据分级、RDS CA 说明以及更可行动的
安装、登录和模型错误提示。源代码合入或推送不代表这些修正已经部署到线上。

## 二、最初目标与完成情况

### 2.1 最初目标

这次工作的出发点不是增加更多页面，而是趁项目仍处于开发期、没有真实用户、
无需保留历史兼容时，解决以下根问题：

- 去除 WordPress 特有概念对 Cloud 核心运行时的无必要绑定。
- 统一站点、账号和用户身份，避免以后依赖邮箱、URL 或可变展示字段作为身份。
- 收敛重复入口、旧合同、兼容别名和第二套控制真相。
- 建立文本和媒体的完整证据链，而不是只证明一次 Provider 调用成功。
- 提升安全、性能、部署可重复性、故障诊断和回滚能力。
- 为未来其他 CMS 和更多媒体操作留下接缝，但不提前制造万能抽象。
- 在没有业务数据的阶段直接切换到外部 PostgreSQL 18，不为无价值的数据迁移
  和双写兼容付费。

## 三、实施历程

| 阶段 | 主要问题 | 核心结果 |
| --- | --- | --- |
| P0：目标合同与基线 | 边界和删除对象不清，容易边改边扩 | 冻结目标合同、证据标准、删除清单和 WordPress-first 原则 |
| P1：身份与运行时基础 | URL、邮箱、WordPress 字段承担了过多身份责任 | 建立不可变主体与站点身份、平台中立连接信封和唯一活动合同 |
| P2：WordPress 文本闭环 | Cloud 调用成功不等于编辑器可审阅、可应用 | 标题、摘要、选中文本改写走正常产品路径，保存仍由 WordPress 明确执行 |
| P3：媒体闭环 | 二进制、交付、ACK、本地写入和回滚责任混杂 | 建立临时制品、签名拉取、完整性校验、传输 ACK、本地采用和回滚链 |
| P4：Portal / Admin 收敛 | 页面可能演变为第二套业务控制面 | 保留服务面、诊断面和只读投影，删除或封闭重复控制真相 |
| P5：硬化与发布 | 本地通过不能证明精确制品、拓扑和恢复可用 | 完成安全、负载、拓扑收缩、exact-bundle、恢复和跨仓库证据 |
| 首次安装与 PG18 | 新主机必须先手工制作复杂 `.env.deploy` | 建立一次性安装、外部 RDS、原子配置、管理员密钥和失败关闭 |
| 操作反馈修正 | 内部字段、笼统报错和证据等级泄露给管理员 | 自动生成内部标识，细化错误，解释 CA，用准确状态描述模型能力 |
| 开发效率修正 | 小改动也运行大门禁，反馈周期失衡 | 把本地验证明确分为功能局部、共享高风险、集成发布三个等级 |

早期阶段追求系统性是合理的，因为身份、写入权、媒体交付和部署恢复会跨越多个
仓库；当前阶段继续以同样强度扩张则不合理，因为关键边界已经获得证据，主要矛盾
已经转为真实使用反馈和迭代速度。

## 四、冻结的产品与架构边界

### 4.1 四层结构

| 层 | 责任 |
| --- | --- |
| Cloud | 模型执行、路由、用量、额度、队列、健康、诊断、临时制品和运行证据 |
| 跨平台运行合同 | 输入输出、版本、幂等、关联 ID、对象引用、错误码和 `suggestion_only` |
| CMS 本地适配器 | 权限、编辑器入口、上下文提取、签名、审阅、应用和本地审计 |
| CMS | 内容、用户、分类、媒体、评论和最终写入真相 |

WordPress、Typecho、Z-BlogPHP、Ghost 是宿主平台；后台编辑器、MCP 或其他入口
是访问渠道。两条轴线不能合并成一种 Adapter，否则会形成“平台 × 渠道”的组合
爆炸。

### 4.2 当前明确不做

- 不为每个 CMS 新建一套 Cloud Runtime 或 `/v1/<cms>/*` 执行体系。
- 不把 abilities、workflow、prompt、审批、发布或本地审计迁入 Cloud。
- 不创建万能 CMS 内容模型或 Cloud 侧能力注册中心。
- 不因未来可能需要而立即抽取跨 CMS SDK。
- 不允许 Cloud 直接保存或发布 WordPress 内容。
- 不用继续“彻底重构”来替代真实体验验证。

未来开始第二个 CMS 时，应先以少量 suggestion-only 任务验证现有接缝，
再根据两个真实实现中重复出现的部分抽取 SDK。CMS 权限、Hook、编辑器 UI、
写入和审计仍留在各自插件。

## 五、媒体能力的长期基础

图片处理是 Cloud 的重要差异化能力，但它的稳定基础不是把更多二进制塞进数据库，
而是一条受治理的媒体作业链：

1. CMS 本地提取输入并取得上传授权。
2. Cloud 接收流式输入，创建带站点边界和幂等键的媒体作业。
3. Provider 或处理器执行操作，Cloud 保存临时制品和完整运行证据。
4. CMS 通过短期签名地址拉取结果并验证类型、大小和摘要。
5. CMS 发送 ACK，证明制品已经成功传输。
6. 用户在本地预览、审阅和选择是否采用。
7. CMS 完成最终写入、引用修复、审计和必要时的回滚。
8. Cloud 按 TTL 和清理策略删除临时制品。

这里有三个不可混淆的事实：

- Provider 生成成功不等于媒体已经交付。
- ACK 成功不等于媒体已经被 WordPress 采用。
- WordPress 采用成功也不授权 Cloud 以后直接修改 CMS。

后续支持音频、视频、文档或更多图片操作时，应复用作业信封、临时制品、
签名交付、完整性校验、ACK、TTL 和证据链；不同媒体的操作参数、预览和本地采用
逻辑由专用操作合同和 CMS 适配器处理。只有在多个真实操作出现稳定共性后，
才扩展共享抽象。

## 六、首次安装与外部 PostgreSQL 18

### 6.1 安装模型

首次安装把应用拆成最小 Bootstrap 和完整 Runtime 两个配置阶段：

- `pending`：只开放经过安装码保护的 Setup、最小存活检查和安装页面。
- `initializing`：持有安装锁，执行数据库验证、迁移和原子配置写入。
- `complete`：加载正式服务，永久关闭 Setup；数据库故障不能让它退回
  `pending`。

安装器要求 PostgreSQL 主版本为 18、空库或同一次中断安装的可恢复状态，
使用 TLS `verify-full` 和 RDS CA，并在所有迁移、配置写入和 Runtime 初始化
成功后才提交 `complete`。生产数据库凭据和根密钥保存在受保护的结构化运行配置，
不作为普通 Admin 设置，也不写入浏览器存储或日志。

### 6.2 两类一次性凭据

- 安装码证明操作者控制服务器，安装完成后失效；遗失时只能在服务器轮换。
- 管理员密钥用于登录 Cloud 管理后台，服务端只保存摘要；遗失时不能找回明文，
  只能在服务器轮换，并使旧密钥和旧会话失效。

真实部署暴露了两个仅靠本地实现难以发现的问题：

1. 管理员轮换脚本对 Compose 容器身份判断过窄，无法识别实际 API 容器。
2. 安装完成后内部 API 主机未进入可信主机集合，导致新密钥虽然生成成功，
   登录请求仍失败。

两个问题均已转化为源代码修复和回归检查，管理员随后确认新密钥可以登录。
这类事实应标记为“操作会话验证”，不能扩大为完整生产稳定性结论。

### 6.3 RDS 连接边界

当前生产合同使用同地域、同 VPC 的私网 RDS 地址和 `verify-full`。CA 用于同时
验证传输加密和服务端身份，不能因安装步骤多而静默降级为不验证证书。

安装页面曾把所有网络不可达都描述为“无法通过私网地址连接”，即使管理员输入的
是公网域名也会产生误导。后续修正将 DNS、私网策略、TCP、TLS、证书和认证阶段
映射为可行动的错误，并在 CA 区域说明证书用途、获取位置和安全要求。

公网 RDS 支持只被记录为未来需求，当前没有实现。若以后确有跨 VPC 或临时验证
需求，必须以明确的高风险操作方式实现：显式选择公网模式、限制白名单、强制
`verify-full`、展示成本和暴露风险，并保留私网为生产默认。不能通过放宽现有
私网校验或把所有地址都当成等价输入来“顺手兼容”。

验证用基础版数据库只适用于开发和小规模 WordPress 体验验证。第一个真实用户、
付费使用或不可替代数据进入前，仍必须升级为高可用实例，并完成备份恢复验证。

## 七、管理员真实反馈带来的修正

### 7.1 内部标识不应要求管理员理解

`connection_id` 是稳定内部标识，不是用户任务。让管理员手工满足“2–64 位小写
字母、数字、点、横线或下划线”会把内部合同泄露到产品界面。

修正原则：

- 新建时由服务端生成不可变标识。
- 管理员填写展示名称、Provider 类型、基础 URL 和凭据。
- 已存在标识只作为高级只读诊断信息。
- 服务端继续校验唯一性，不能把安全责任转移到前端。

### 7.2 “目录可见”不等于“运行可用”

模型可以处于三个不同证据等级：

1. 目录已发现：Provider 声明或列表接口可见。
2. 已配置：模型被选择并绑定到运行配置。
3. 已验证可执行：受控探测或真实运行成功。

界面必须使用准确状态，不能因为模型出现在目录里就宣称可运行，也不能在缺少成功
证据时自动启用。失败提示应告诉管理员下一步检查凭据、基础 URL、模型 ID、能力
类型还是 Provider 响应。

### 7.3 错误消息必须帮助行动

外部错误信封保留稳定错误码，中文界面根据错误码给出可执行建议；内部异常、
数据库 URL、密码、CA 内容、解析后的地址和 Provider 原始敏感响应保持脱敏。

重点不是把错误全文翻译成中文，而是回答三个问题：

- 哪一步失败？
- 管理员可以检查什么？
- 哪些细节可安全提供给支持人员？

## 八、工作审视报告

### 8.1 做得好的部分

- 在没有真实用户时直接删除旧合同和兼容别名，避免把临时设计永久化。
- 先冻结 Cloud 与 WordPress 的写入和治理边界，再并行推进实现。
- 用唯一身份、幂等、关联 ID、签名交付和证据链处理跨仓库问题。
- 把文本“返回结果”和媒体“成功交付、本地采用、回滚”分别验收。
- 将本地/CI、exact-bundle、受控生产和真实用户证据明确分层。
- 生产代码不在服务器直接修改；部署中发现的问题回到 Git 修复。
- 在用户工作树存在未提交内容时使用干净临时工作树，并精确暂存任务文件。
- 实际错误不只临时绕过，而是转化为错误分类、回归检查和文档规则。

### 8.2 问题、根因和改进

| 观察到的问题 | 根因 | 影响 | 已采取或后续改进 |
| --- | --- | --- | --- |
| 小范围管理员体验修正仍运行完整合同和领域测试 | `check:fast` 名称像快速反馈，但实际是集成门禁；规范未区分风险等级 | 单次反馈周期被不必要地拉长，抑制小步迭代 | 已建立 Tier 1/2/3 风险分级，局部改动默认只跑聚焦检查 |
| 数据库不可达提示始终写“私网” | 后端把多阶段连接错误折叠为单一错误投影 | 管理员即使输入公网地址也无法判断真实失败环节 | 分类 DNS、策略、TCP、TLS、证书、认证和数据库状态，前端给行动建议 |
| `connection_id` 要管理员手工满足英文格式 | 内部资源标识直接暴露为业务输入 | 增加认知负担，产生无业务价值的表单错误 | 服务端生成，界面隐藏或只读展示 |
| 模型目录把“可见”误表达为“可执行” | 混合了发现、配置和运行证据 | 容易让管理员错误启用并难以排查 | 固定三档证据状态，不在缺少运行证据时宣称可用 |
| 管理员密钥轮换和登录在真实部署失败 | Compose 容器身份与可信主机属于环境接缝，本地测试没有覆盖实际形态 | 密钥生成后仍无法进入后台 | 修复完整容器身份比较和安装后内部主机信任，并保留回归检查 |
| 干净工作树复用依赖或环境时可能读到其他工作区状态 | 测试环境隔离和便利性之间缺少明确边界 | 结果可能受本机环境或秘密配置污染 | 干净工作树只复用依赖缓存，不复用秘密配置；关键检查显式隔离环境 |
| 系统性检查持续时间过长 | 重构阶段的高安全标准被沿用到所有小改动 | 质量较高，但迭代速度与当前无用户阶段不匹配 | 结构性重构正式停止，验证成本按变更风险而不是心理不安升级 |
| 架构讨论容易扩展到多 CMS 和所有媒体 | “为未来打基础”容易被误解为“现在就实现未来” | 增加抽象和测试矩阵，延迟 WordPress 体验 | 当前只保留接缝；第二个真实实现出现后再抽共享部分 |

### 8.3 核心经验

1. **先确定真相归属，再设计接口。**
   跨系统最危险的不是字段少，而是两边都以为自己拥有最终真相。

2. **基础能力应由第二个真实实现证明，而不是由第一个实现想象。**
   现在可以定义中立接缝，但不能凭空构造万能 CMS 或万能媒体模型。

3. **没有用户时应主动删除兼容成本，但不能删除安全边界。**
   旧字段、旧路由、双写和别名可以直接移除；TLS、权限、幂等、原子写入、
   脱敏和回滚不能以“开发期”为理由省略。

4. **运行成功、交付成功、用户采用和业务价值是四个不同层次。**
   Provider 返回 200 只能证明最内层，不能自动升级为产品完成。

5. **真实操作问题优先于继续架构推演。**
   安装、密钥轮换、登录、Provider 配置和 WordPress 往返暴露的问题，
   比再增加一层抽象更有价值。

6. **测试强度必须匹配风险。**
   多跑测试不是免费的“更严谨”；无差别全量验证会减少迭代次数，反而降低
   真实问题被及时修复的概率。

7. **源代码、部署制品和线上状态必须分开陈述。**
   提交、推送、合并、构建、部署和线上 Smoke 是六个独立事实。

## 九、后续高效开发方法

### 9.1 每批工作先写变更信封

在修改前只回答必要问题：

- 本批解决哪个真实问题？
- 哪个模块拥有它？
- 明确不做什么？
- 哪个公开合同会变化？
- 哪些文件可改，哪些区域不可改？
- 最窄有效验证是什么？
- 如何回滚？

一个批次只处理一个主要矛盾。若多个实现可以并行，应按文件所有权切分：
主代理负责合同、边界、集成审查、验收和 Git；子代理各自负责一个不重叠的文件
区域及初步测试。多个代理需要同时修改同一核心文件时，停止并行，由主代理串行
集成。

### 9.2 使用三档验证

| 等级 | 适用范围 | 默认验证 |
| --- | --- | --- |
| Tier 1：功能局部 | 文案、局部 UI、单个适配器、文档、已有合同内的小修 | 聚焦测试、类型检查或静态合同；默认不跑 `check:fast` |
| Tier 2：共享或高风险接缝 | API/错误合同、认证、配置、数据库、Worker、依赖、安全、共享模型 | 聚焦测试加对应的 seam、perimeter、anti-drift 或静态门禁 |
| Tier 3：集成与发布 | 多模块里程碑、跨仓库合同、发布候选、生产部署 | `check:fast`、所需完整门禁、矩阵、exact-bundle 和部署 Smoke |

升级验证的理由必须来自变更触达范围、不可逆性和失败影响，而不是“改了代码就
全部跑一遍”。CI 仍保留集成级兜底；本地开发优先获得快速、可信的第一反馈。

### 9.3 采用停止规则

满足以下条件后，当前问题应关闭而不是继续优化：

- 用户可见问题已经消失。
- 公共合同没有产生新的歧义。
- 聚焦回归可以稳定重现并保护修复。
- 没有越过 Cloud / WordPress 边界。
- 剩余改进只有假设，没有真实失败或用户反馈。

只有出现新的边界失败、数据风险、真实用户阻塞或第二个实现证据时，才重新打开
架构问题。

## 十、下一阶段

### P0：关闭 CVE 发布门槛

当前权威收尾记录仍保留三项受控 Python 3.14.6 High 例外，最迟于
`2026-08-05` 失效。应以最新精确镜像扫描为依据消除条目；不能自动延期，也不能
把受控验证例外解释为 GA 授权。

### P1：发布当前管理员体验修正

在下一次有计划的部署中合入、构建和验证本分支修正，不因本次文档提交立即更新
服务器。最小 Smoke 包括：

- 已完成安装的实例不会重新开放 `/setup`。
- 新管理员密钥可以登录，旧密钥和旧会话在轮换后失效。
- Provider 新建不要求管理员填写 `connection_id`。
- 模型目录区分已发现、已配置和已验证可执行。
- RDS CA 说明和数据库错误不泄露秘密，并能指导下一步操作。

### P2：小规模 WordPress 体验验证

只选择当前最有代表性的文本和图片任务，记录：

- 任务成功率、失败阶段和重试结果。
- 首次可用时间和端到端延迟。
- 建议被接受、修改或放弃的比例。
- 图片预览、采用、引用修复和回滚是否自然。
- 管理员需要查看日志或求助的次数。
- Cloud 资源、RDS 连接和 Worker 是否稳定。

当前没有真实用户，因此不能预设这些指标已经达标。先观察少量真实工作流，
再决定优化模型、提示、交互、容量还是诊断。

### P3：做一次明确的 GA 决策

只有在 CVE、数据库高可用、备份恢复、WordPress 体验和发布回滚证据齐备后，
才决定是否接入第一个真实用户。没有通过时保持受控验证状态，不用继续堆功能来
掩盖发布门槛。

### 明确延期

- 公网 RDS 连接及其风险控制。
- Typecho、Z-BlogPHP、Ghost 插件。
- 通用 PHP SDK。
- 音频、视频、文档等新媒体类型。
- 多管理员、注册、找回密码和 MFA。
- 自动发布、批量写入和 Cloud 直接写 CMS。

延期事项应由真实需求重新立项，不能在 WordPress 体验批次里顺带实现。

## 十一、收尾判定

本轮重构的最终目标不是得到“最专业、最完整的架构”，而是得到一个边界清晰、
能安全安装、可重复部署、能完成 WordPress 文本与图片闭环、可以快速继续验证的
Cloud 基础。

这个目标已经达到工程收尾条件。接下来最有价值的工作是缩短反馈环、解决真实
操作问题、观察 WordPress 使用价值，并在证据充分后做 GA 或扩平台决策。
