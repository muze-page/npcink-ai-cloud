# ADR-003: Use `principal_id` As The Stable Cloud User Identity

## Status

Accepted.

## Date

2026-07-14.

## Context

Cloud already had a commercial identity model centered on `principals`, account
memberships, provider bindings, Portal sessions, trial claims, support records,
and audit evidence. The migration
`migrations/versions/20260627_0048_principal_identity_contract.py` established
the core tables and constraints, but the identity hierarchy was not stated
clearly enough for future development.

That ambiguity created several drift risks:

- treating a mutable email address as the permanent user key;
- introducing parallel identifiers such as `user_id` or `customer_id`;
- confusing a commercial account or membership with a natural person;
- allowing a provider binding to move silently between principals;
- exposing internal Cloud identifiers in the customer Portal;
- reviving `site_admin` as a third product identity type;
- treating a WordPress user reference as Cloud-wide identity truth.

The closeout therefore began with an evidence-first audit of the model,
repository, Portal and admin services, migration, frontend client, UI contracts,
and API tests. It preserved capabilities that already satisfied the contract
and changed only confirmed gaps. The implementation landed in commit
`44afe8e5` (`Freeze stable Cloud principal identity`).

## Historical Findings

The audit confirmed that most of the required foundation already existed:

- `Principal.principal_id` was the primary key;
- the server generated IDs as `prn_<uuid4 hex>`;
- repeated registration with the same email reused the existing principal;
- email changes updated the login alias without replacing the principal;
- account memberships referenced a principal and an account independently;
- provider subjects lived in `identity_provider_bindings`;
- Portal sessions, support, audit, and trial evidence referenced
  `principal_id`;
- the provider and external-subject pair had a database uniqueness constraint;
- the customer account page already avoided internal identifier disclosure;
- the admin user directory already returned and searched principal IDs.

Four real gaps remained:

1. The README did not explain the internal identity hierarchy.
2. The repository upsert path could reassign an existing provider binding to a
   different principal.
3. The frontend product identity union still included the legacy
   `site_admin` value.
4. The admin user inspector showed the full principal ID but did not provide a
   direct copy action for support and audit work.

No new user table, identity column, authentication framework, role system, or
migration was required.

## Decision

1. `principals.principal_id` is the single stable and immutable Cloud user
   identity.
2. The ID is generated by the server in the existing `prn_<uuid4 hex>` format.
3. Changing email, binding or unbinding a provider, joining another account,
   changing sites, or changing package must not replace the principal ID.
4. Email is a mutable login alias. Provider subjects such as QQ OpenID or
   UnionID are external login aliases stored only in
   `identity_provider_bindings` and mapped to a principal.
5. `account_id` identifies a commercial account or tenant.
   `membership_id` identifies the relationship between a principal and an
   account. Neither identifies the user.
6. A site is another independent resource dimension. A WordPress `wp_user_id`
   is only a site-scoped external reference and never Cloud global identity or
   permission truth.
7. Product identity semantics remain limited to `platform_admin` and `user`.
   Permission differences must be expressed through bounded capabilities, not
   additional identity labels.
8. Portal sessions and internal commercial evidence may carry `principal_id`,
   but the customer `/portal/account` surface must not display
   `principal_id`, `account_id`, internal roles, or backoffice support markers.
9. The administrator `/admin/portal-users` surface may display, copy, and
   search the full principal ID for audit, support, and operational diagnosis.
10. If customers later need a reference they can quote to support, it must be a
    separately designed support reference rather than disclosure or renaming
    of `principal_id`.

## Implementation Approach

The closeout follows these development principles:

### Evidence Before Redesign

Inspect migrations, models, mutation paths, API payloads, UI behavior, and tests
before proposing schema work. Existing correct behavior is evidence to retain,
not a reason to rewrite the system.

### Enforce Invariants At Mutation Boundaries

The repository refuses to change the principal of an existing provider or
UnionID binding. Database uniqueness remains the final guard for the provider
and external-subject pair. Tests assert the frozen ID format and binding
immutability.

### Keep Resource Dimensions Separate

Principal, account, membership, site, subscription, and WordPress user
references answer different questions. APIs and UI code must not collapse them
into a generic `user_id` or infer one from another.

### Separate Internal Truth From Customer Presentation

Internal sessions and services retain stable identity evidence. Customer UI
shows understandable account controls without exposing implementation IDs.
Backoffice UI exposes the exact identifiers operators need and protects that
behavior with contract tests.

### Prefer The Smallest Reviewable Change

The closeout supplemented the existing README, hardened one repository path,
corrected one frontend type, added one admin action, and expanded focused tests.
Identity hunks were reviewed and staged separately from concurrent Site
Knowledge and usage work.

## Alternatives Considered

### Use Email As The User Key

Rejected because email is mutable, may be recovered or reassigned, and is also
personally identifiable login data. It cannot provide durable joins for audit,
support, trials, and provider bindings.

### Add A New `user_id` Or Customer Directory

Rejected because `principal_id` already provides the required stable identity.
A parallel directory would create synchronization, migration, and ownership
ambiguity and could turn Cloud into a second WordPress control plane.

### Rename `principal_id`

Rejected because the existing name is already used across models, migrations,
sessions, commercial evidence, and tests. Renaming would add compatibility work
without improving the identity model.

### Expose The Internal ID To Customers

Rejected because the internal key is useful for backoffice correlation, not a
customer-facing concept. A future support reference should have its own
contract, lifecycle, and disclosure policy.

## Verification Evidence

The closeout added or confirmed coverage for:

- repeated registration with the same email reusing one principal;
- email changes preserving the principal ID;
- provider binding and unbinding preserving the principal ID;
- one principal holding memberships in multiple accounts;
- provider and external-subject uniqueness;
- admin lookup and copy behavior for principal IDs;
- customer account UI hiding internal identifiers;
- product identity types remaining `platform_admin | user`.

The focused Portal API, service API, frontend contract, lint, type-check, and
repository `check:fast` gates passed at closeout. Skipped tests and the existing
Starlette and pnpm configuration warnings were reported separately and were not
caused by the identity change.

## Consequences

- Future authentication providers must resolve to an existing or newly created
  principal; they must not create a new category of Cloud user identity.
- Commercial and support features should join on `principal_id`, while account
  scope remains explicit through account or membership references.
- Customer UI contracts must continue to prevent accidental identifier
  disclosure.
- Admin workflows may use the full principal ID as an exact operational key.
- No WordPress user, role, or permission ownership moves into Cloud.

The provider and external-subject pair is protected by a database unique
constraint. UnionID conflicts are rejected in the service and repository
layers, but there is no separate database uniqueness constraint for UnionID.
This avoids prematurely forbidding legitimate multi-subject mappings; if real
collision evidence appears, the provider scoping and concurrency policy must be
designed before adding a migration.

## Maintenance Rules

When changing authentication or commercial identity code:

1. Search for mutations of `principal_id` and reject reassignment paths.
2. Add aliases through provider bindings rather than new principal columns.
3. Keep account, membership, site, and WordPress references explicit.
4. Update both admin visibility and customer non-disclosure contract tests when
   changing identity presentation.
5. Do not add product identity values beyond `platform_admin` and `user`.
6. Add a migration only when an actual storage invariant cannot be enforced by
   the existing schema and mutation boundaries.

## Rollback

This ADR documents an already accepted and implemented contract. Reverting the
document alone does not revert the runtime behavior. Any future replacement of
the identity contract requires a superseding ADR, an explicit data migration,
compatibility analysis for sessions and commercial evidence, and an unchanged
Cloud-versus-WordPress ownership boundary.
